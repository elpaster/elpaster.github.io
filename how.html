How to Build Accessible Sites

A comprehensive guide to creating inclusive web experiences

Building Accessibility from the Ground Up

Creating accessible websites requires understanding both the technical foundations and design principles that make content usable for everyone. This guide will walk you through the essential techniques, from planning your information architecture to testing your final product.

account_treeInformation Architecture

What is Information Architecture?

Information Architecture (IA) is the structural design of information—how content is organized, labeled, and navigated. Good IA is the foundation of an accessible website because it helps all users find what they need quickly and efficiently.

Key Principles of Accessible IA

1. Clear Hierarchy

Organize content in a logical tree structure with clear parent-child relationships. Users should always understand where they are and how to get to related content.

✓ Good Example:

Home
├── Products
│   ├── Category A
│   │   ├── Product 1
│   │   └── Product 2
│   └── Category B
└── About Us

2. Consistent Navigation

Keep navigation menus in the same location and order across all pages. This helps users build mental models of your site structure.

✓ Good Example:

Navigation appears in the same position on every page with identical labels and order.

✗ Bad Example:

Different pages have navigation in different positions or with varying labels for the same destination.

3. Descriptive Labels

Use clear, descriptive names for navigation items that accurately describe the destination.

✓ Good Example:

"Contact Us" | "Product Catalog" | "Customer Support"

✗ Bad Example:

"Click Here" | "More" | "Go"

4. Multiple Ways to Navigate

Provide different navigation methods: main menu, search, breadcrumbs, and sitemaps. Users have different preferences and needs.

  • Main Navigation: Primary menu for common destinations
  • Search: For users who know what they want
  • Breadcrumbs: Show current location in hierarchy
  • Sitemap: Overview of entire site structure

5. Limit Menu Depth

Keep navigation menus shallow (3 levels maximum). Deep nested menus are difficult to navigate, especially for keyboard and screen reader users.

codeSemantic HTML Best Practices

What is Semantic HTML?

Semantic HTML means using HTML elements according to their intended purpose. Instead of using generic <div> and <span> tags for everything, semantic HTML uses meaningful elements like <header>, <nav>, <article>, and <footer>.

Why it matters: Screen readers and other assistive technologies rely on semantic HTML to understand page structure and navigate content effectively.

Document Structure Tags

✓ Good Example:

<header>
  <nav>
    <ul>
      <li><a href="/">Home</a></li>
    </ul>
  </nav>
</header>

<main>
  <article>
    <h1>Page Title</h1>
    <section>
      <h2>Section Title</h2>
      <p>Content here</p>
    </section>
  </article>
  <aside>
    <h2>Related Links</h2>
  </aside>
</main>

<footer>
  <p>Copyright 2025</p>
</footer>

✗ Bad Example:

<div class="header">
  <div class="nav">
    <div class="menu">
      <a href="/">Home</a>
    </div>
  </div>
</div>

<div class="content">
  <div class="title">Page Title</div>
  <div class="text">Content here</div>
</div>

This "div soup" provides no semantic meaning to assistive technologies.

Heading Hierarchy

Use headings (h1-h6) in proper order without skipping levels. Screen readers use headings to navigate and understand content structure.

✓ Good Example:

<h1>Main Page Title</h1>
<h2>First Major Section</h2>
<h3>Subsection</h3>
<h3>Another Subsection</h3>
<h2>Second Major Section</h2>

✗ Bad Example:

<h1>Main Page Title</h1>
<h3>Section</h3> <!-- Skipped h2 -->
<h5>Subsection</h5> <!-- Skipped h4 -->

Skipping heading levels confuses screen reader users about content hierarchy.

Lists and Navigation

✓ Good Example:

<nav aria-label="Main navigation">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/about">About</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>

Screen readers announce "Navigation, list of 3 items" and let users jump between list items.

✗ Bad Example:

<div class="nav">
  <a href="/">Home</a>
  <a href="/about">About</a>
  <a href="/contact">Contact</a>
</div>

No semantic structure—screen readers can't easily identify this as navigation.

Forms and Labels

✓ Good Example:

<form>
  <label for="email">Email Address</label>
  <input type="email" id="email" 
         name="email" required 
         aria-describedby="email-hint">
  <span id="email-hint">
    We'll never share your email
  </span>
  
  <button type="submit">Subscribe</button>
</form>

Screen readers announce the label and hint when the field receives focus.

✗ Bad Example:

<form>
  Email Address
  <input type="text" placeholder="Email">
  <div>Click to subscribe</div>
</form>

No label association, placeholder disappears when typing, and the button isn't actually a button.

Buttons vs Links

✓ Good Example:

<!-- Use buttons for actions -->
<button type="submit">Submit Form</button>
<button onclick="openModal()">Open Dialog</button>

<!-- Use links for navigation -->
<a href="/about">About Us</a>
<a href="/products">View Products</a>

✗ Bad Example:

<!-- Don't use links as buttons -->
<a href="#" onclick="submitForm()">Submit</a>

<!-- Don't use divs as buttons -->
<div onclick="openModal()">Open</div>

Screen readers expect buttons to perform actions and links to navigate. This breaks user expectations.

Images and Alt Text

✓ Good Examples:

<!-- Informative image -->
<img src="chart.png" 
     alt="Bar chart showing 40% increase in sales from 2023 to 2024">

<!-- Functional image (link) -->
<a href="/">
  <img src="logo.png" alt="Company Name Home">
</a>

<!-- Decorative image -->
<img src="decoration.png" alt="" role="presentation">

✗ Bad Examples:

<!-- Generic alt text -->
<img src="chart.png" alt="image">

<!-- Missing alt text -->
<img src="logo.png">

<!-- Alt text on decorative image -->
<img src="decoration.png" 
     alt="blue wavy line decoration">

Alt text should be descriptive for meaningful images, empty for decorative ones.

labelHTML Tags and Classes Best Practices

Using Classes Effectively

Classes should describe purpose or content type, not appearance. This makes code more maintainable and accessible.

✓ Good Class Names (Semantic):

<article class="blog-post">
<button class="primary-action">
<nav class="main-navigation">
<aside class="related-content">
<div class="error-message">

These names describe what the element is or does, making code understandable.

✗ Bad Class Names (Presentational):

<article class="blue-box">
<button class="big-red-button">
<nav class="top-bar">
<aside class="right-sidebar">
<div class="bold-text">

These describe appearance, which becomes misleading if design changes.

Essential Accessibility Attributes

lang attribute

Specify the language of your content so screen readers use correct pronunciation.

<html lang="en">
<p lang="es">Hola, mundo!</p>

aria-label

Provide accessible names for elements without visible text.

<button aria-label="Close dialog">
  <span aria-hidden="true">×</span>
</button>

<nav aria-label="Main navigation">...</nav>

aria-describedby

Link elements to descriptive text.

<input type="password" 
       aria-describedby="password-requirements">
<div id="password-requirements">
  Must be at least 8 characters
</div>

aria-live

Announce dynamic content changes to screen readers.

<div aria-live="polite" aria-atomic="true">
  <!-- Status messages appear here -->
  Form submitted successfully!
</div>

role attribute

Define the purpose of an element when semantic HTML isn't enough.

<div role="alert">Error: Please try again</div>
<div role="search">...</div>

Note: Only use ARIA roles when semantic HTML elements can't achieve the same result.

paletteAccessible CSS: Color and Contrast

Understanding Color Contrast

Color contrast is the difference in brightness between text and its background. Adequate contrast ensures text is readable for people with low vision or color blindness.

WCAG Contrast Requirements

Text Size Level AA (Minimum) Level AAA (Enhanced)
Normal text (under 18pt/24px) 4.5:1 7:1
Large text (18pt+ or 14pt+ bold) 3:1 4.5:1
UI components & graphics 3:1 Not required

Building an Accessible Color Palette

✓ Good Contrast Examples:

Black on White
Contrast: 21:1 (Excellent)
Dark Blue on White Background
Contrast: 7.4:1 (AAA Level)
Nearly Black on White
Contrast: 16.1:1 (Excellent)

✗ Poor Contrast Examples:

Light Gray on White
Contrast: 1.6:1 (Fails all levels)
Red on Pink
Contrast: 1.9:1 (Fails all levels)
White on Yellow
Contrast: 1.2:1 (Fails all levels)

CSS Best Practices for Accessibility

1. Never Remove Focus Indicators

✗ Bad:

/* Never do this! */
* {
  outline: none;
}

a:focus, button:focus {
  outline: 0;
}

Removing focus indicators makes keyboard navigation impossible.

✓ Good:

/* Always provide visible focus */
a:focus, button:focus {
  outline: 2px solid #2d5a7b;
  outline-offset: 2px;
}

/* Or use modern focus-visible */
button:focus-visible {
  outline: 3px solid #2d5a7b;
  outline-offset: 2px;
}

2. Use Relative Units

✓ Good:

body {
  font-size: 100%; /* Respects user preferences */
}

h1 {
  font-size: 2rem; /* Scales with base font size */
}

.container {
  max-width: 60ch; /* Based on character width */
  padding: 1em;
}

Relative units allow users to adjust text size in browser settings.

✗ Bad:

body {
  font-size: 16px; /* Fixed size */
}

h1 {
  font-size: 32px; /* Doesn't scale */
}

3. Don't Rely on Color Alone

✓ Good:

.error {
  color: #d32f2f;
  border-left: 4px solid #d32f2f;
}

.error::before {
  content: "⚠ ";
}

.success {
  color: #388e3c;
  border-left: 4px solid #388e3c;
}

.success::before {
  content: "✓ ";
}

Icons and borders supplement color to convey meaning.

4. Ensure Sufficient Touch Targets

✓ Good:

button, a {
  min-height: 44px;
  min-width: 44px;
  padding: 0.5rem 1rem;
}

/* Add space between interactive elements */
nav a {
  margin: 0.5rem;
}

44×44 pixels minimum makes targets easy to tap on mobile devices.

5. Use CSS Variables for Consistent Colors

✓ Good:

:root {
  --color-primary: #2d5a7b;
  --color-text: #1a1a1a;
  --color-background: #ffffff;
  --color-error: #d32f2f;
  --color-success: #388e3c;
}

body {
  color: var(--color-text);
  background: var(--color-background);
}

.button-primary {
  background: var(--color-primary);
  color: var(--color-background);
}

CSS variables make it easy to maintain consistent, accessible color schemes.

Tools for Checking Contrast

  • WebAIM Contrast Checker: webaim.org/resources/contrastchecker
  • Chrome DevTools: Built-in contrast ratio tool in the color picker
  • Accessible Colors: accessible-colors.com
  • Contrast Ratio: contrast-ratio.com

keyboardKeyboard Accessibility

Why Keyboard Accessibility Matters

Many users rely on keyboards instead of mice, including people with motor disabilities, blind users with screen readers, and power users who prefer keyboard shortcuts. Every interactive element must be usable with only a keyboard.

Essential Keyboard Navigation Patterns

Tab Navigation

  • Tab: Move forward through interactive elements
  • Shift + Tab: Move backward through interactive elements
  • Enter/Space: Activate buttons and links

✓ Good Example:

<button>Submit</button>
<a href="/page">Link</a>
<input type="text">

All these elements are naturally keyboard accessible.

✗ Bad Example:

<div onclick="submit()">Submit</div>
<span onclick="navigate()">Link</span>

Divs and spans aren't in tab order and don't respond to keyboard events.

Skip Links

Skip links allow keyboard users to bypass repetitive navigation and jump directly to main content.

✓ Implementation:

<!-- HTML -->
<a href="#main-content" class="skip-link">
  Skip to main content
</a>

<main id="main-content">
  <!-- Page content -->
</main>
/* CSS */
.skip-link {
  position: absolute;
  top: -100px;
  left: 20px;
  background: #2d5a7b;
  color: white;
  padding: 12px 24px;
  text-decoration: none;
  border-radius: 4px;
  z-index: 1000;
}

.skip-link:focus {
  top: 20px;
}

Skip link is hidden until keyboard focus, then appears at top of page.

Managing Focus

When content changes dynamically (modals, menus, tabs), focus must be managed programmatically.

✓ Modal Dialog Example:

// When modal opens
function openModal() {
  modal.style.display = 'block';
  modal.querySelector('.close-button').focus();
  // Trap focus inside modal
  document.addEventListener('keydown', trapFocus);
}

// When modal closes
function closeModal() {
  modal.style.display = 'none';
  openButton.focus(); // Return focus
  document.removeEventListener('keydown', trapFocus);
}

Arrow Key Navigation

For complex components like tabs, menus, and toolbars, implement arrow key navigation.

✓ Tab Panel Example:

<div role="tablist" aria-label="Sample Tabs">
  <button role="tab" aria-selected="true" 
          aria-controls="panel1">
    Tab 1
  </button>
  <button role="tab" aria-selected="false" 
          aria-controls="panel2">
    Tab 2
  </button>
</div>

<div role="tabpanel" id="panel1">
  Content 1
</div>

Left/Right arrows move between tabs, Tab key moves to panel content.

Comprehensive Testing Approach

Accessibility testing requires multiple methods: automated tools catch common issues, but manual testing finds nuanced problems that tools miss.

1. Automated Testing Tools

These tools scan your code for accessibility violations. Use multiple tools as each catches different issues.

WAVE Browser Extension

What it does: Visual feedback showing accessibility errors and warnings directly on your page

Best for: Quick visual inspection, identifying missing alt text, contrast issues

Link: wave.webaim.org/extension

How to use:
  1. Install WAVE extension for Chrome/Firefox
  2. Navigate to your webpage
  3. Click WAVE icon in browser toolbar
  4. Review errors (red), alerts (yellow), and features (green)

Lighthouse

What it does: Built-in Chrome tool that audits accessibility, performance, and more

Best for: Overall accessibility score and key metrics

How to use:
  1. Open Chrome DevTools (F12)
  2. Go to "Lighthouse" tab
  3. Check "Accessibility" category
  4. Click "Analyze page load"
  5. Review score and failing audits

HTML Validator

What it does: Checks for valid HTML structure

Best for: Finding markup errors that affect accessibility

Link: validator.w3.org

Automated tools catch about 30-40% of accessibility issues. Manual testing is essential.

3. Screen Reader Testing

Testing with screen readers reveals how blind users experience your site.

NVDA (Windows - Free)

Download: nvaccess.org

Basic Commands:
  • NVDA + N: Open NVDA menu
  • NVDA + Down Arrow: Read next item
  • H: Next heading
  • K: Next link
  • B: Next button
  • F: Next form field

VoiceOver (Mac/iOS - Built-in)

Activate: Cmd + F5 (Mac)

Basic Commands:
  • VO + Right Arrow: Next item
  • VO + Cmd + H: Next heading
  • VO + U: Open rotor (navigation menu)
  • VO + Space: Activate item

What to Test With Screen Readers:

  • Can you understand page structure from headings?
  • Are landmarks announced correctly (navigation, main, complementary)?
  • Do images have meaningful alt text read aloud?
  • Are form fields properly labeled?
  • Do buttons/links have clear purposes?
  • Are dynamic changes announced?
  • Can you complete tasks without seeing the screen?

4. Mobile and Responsive Testing

Touch Target Size

All interactive elements should be at least 44×44 pixels

Portrait and Landscape

Content should work in both orientations

Zoom and Reflow

Content should reflow without horizontal scrolling when zoomed

Mobile Screen Readers

Test with VoiceOver (iOS) or TalkBack (Android)

5. Browser Testing

Test in multiple browsers and platforms:

  • Chrome (most popular)
  • Firefox (excellent accessibility support)
  • Safari (important for Mac/iOS users)
  • Edge (important for Windows users)

6. User Testing

Testing with People with Disabilities

The most valuable testing comes from real users. Considerations:

  • Recruit diverse users (different disabilities, assistive technologies)
  • Compensate participants fairly for their time
  • Observe without interrupting their natural workflow
  • Ask about frustrations and what works well
  • Test early and often—not just at the end
